NOTE ON MEASUREMENT CONDITIONS
-------------------------------
All 3 runs below were taken back-to-back with an unrelated user process
(`scripts/bench5k.py`, a Python script, ~99% of one core) running
concurrently in the background. This was NOT a quiet machine. The 3 runs
are recorded verbatim, without cherry-picking, specifically because they
disagree with each other:

  peak differential sync overhead per run: 2.70%  / 14.74% / 6.44%
  binary's own verdict per run:            PASS   / PARTIAL / PARTIAL

This is NOT a stable "<5%" result — one run passes the binary's own 5%
threshold, two do not. The peak figure is noise-dominated at this thread
count on this hardware under contention; no single number here should be
read as "the" measured sync overhead. A quieter machine and/or more reps
would be needed to get a stable per-run figure.

A second, separate anomaly across all 3 runs: the transposition-table (TT)
differential is consistently strongly NEGATIVE (contended arm faster than
partitioned, roughly -46% to -80%), which is the opposite of what the
"sync overhead" metric is designed to detect (it assumes partitioned >=
contended). This is reproducible across all 3 runs (not one-off noise) but
its cause is not understood here — possibly strip-size/prefetch effects in
the partitioned access pattern at this TT size, not investigated further.
Only the history-table sub-benchmark's numbers were used for the
"peak differential sync overhead" figures above, matching what the binary
itself reports as the max of (tt_peak, hist_peak) — but given TT's negative
peak never becomes the max, the binary's own peak-overhead figure never
actually reflects the TT anomaly. Worth investigating before trusting the
TT half of this tool.

WHAT THIS DOES AND DOES NOT VERIFY
------------------------------------
AGENTS.md's DoD states: "Thread synchronization overhead must remain below
5% even when running on 64+ cores." This measurement is taken on 10 cores.
It does NOT verify the 64+-core claim — no such hardware was available.
The architecture argument (all shared tables use Relaxed atomics, no CAS,
no blocking, so overhead is purely cache-line-bouncing cost) is a plausible
reason to expect similar behavior at higher core counts, but that is the
codebase's argument, not something demonstrated here — cache-line
contention often gets *worse*, not better, as core count increases, so it
should not be assumed to extrapolate favorably. The 64+-core DoD remains
unverified for lack of hardware.

Hardware threads : 10
Metric           : differential (contended vs partitioned), best of 7 reps

TT size          : 4 MiB

P/E core heterogeneity and Amdahl effects cancel by design:

       1       -55.08%
       2       -49.20%
       4       -46.88%
       8       -28.97%
      10       -65.56%

--- History table (2 268 × AtomicI32, 97% load / 3% store) ---
 threads  sync overhead
       1         0.81%
       2         0.15%
       4       -14.90%
       8         2.08%
      10         2.70%

Peak differential sync overhead: 2.70%

PASS  sync overhead 2.70% < 5%

Architecture:
  All shared tables use Relaxed load/store/fetch_add — no CAS, no blocking.
  Killer table: one 64-byte cache line per ply (padded), zero inter-ply
  false sharing (not benchmarked here; overhead is by construction 0%
  when threads search distinct plies, as in YBW parallel search).

Note: measured on 10 cores (Apple M4: 4 P + 6 E).  The DoD
specifies '64+ cores (verified by profiler)' — that requires 64-core
hardware.  On a homogeneous server the measured overhead is a strict
upper bound because cache lines per thread increase with core count.
